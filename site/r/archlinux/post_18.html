It is not, though. The myth that Arch is "minimal" is because it comes naked. If you install a full desktop environment it is not more or less lightweight than an Ubuntu installation with the same DE. Compare a base Arch install with an Debian/Fedora/Ubuntu minimal installation and you won't find much difference in terms of resource usage.

Actually most other distributions offer split packages for most/all of their packages (e.g. pkg-dev or pkg-lib) while in Arch you often have to install the full package just to use the library because upstream only offers full packages and Arch tries to be as close to upstream as possible. So in term of disk usage Arch is usually heavier than other distributions.

kimiho_mir 4

You're completely missing the point here by confusing disk space and static RAM allocation with actual CPU activity. Saying a 'minimal' Ubuntu or Mint install has the same resource usage as Arch is fundamentally misunderstanding how these distros operate under the hood. Ubuntu/Mint 'minimal' is just their standard heavy base with some GUI apps like LibreOffice stripped out. The backend is exactly the same. You still have snapd mounting a dozen loop devices, unattended-upgrades polling servers, apport running for crash telemetry, and various auto-updaters hanging around. All of these default background services constantly wake up an older CPU like the i7-2630QM from deep C-states. That constant background polling and interrupt handling is exactly what generates heat and keeps the fans spinning at idle. Arch isn't minimal just because it 'comes naked' — it's minimal because of its bottom-up, opt-in architecture. When you install a DE on Arch, you get the pure, vanilla upstream code. No distribution-patched bloat, no forced telemetry, and absolutely ZERO background polling services running unless you explicitly systemctl enable them yourself.

Fewer background daemons = fewer CPU wakeups = lower idle temps. It really is that simple
