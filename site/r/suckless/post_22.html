i got the suckless core (sinit, ubase, sbase, smdev...) working on an alpine vm

by stvpidcvnt111111 | 63 points

https://v.redd.it/irk90an5elqg1

i did dash as /bin/sh, mksh as interactive shell, onetrueawk, replaced busybox with ubase/sbase, sdhcp as dhcpclient, smdev instead busybox mdev, svc and sinit instead of openrc, and i also used the morpheus init scripts.

i did add the svc scripts but i didnt add them to start on boot so that is part of the reason why the boot is so fast.

i mostly did this as a sort of proof-of-concept because i want to make my own alpine-based distro using the suckless core, altho as you can probably tell by the video i still have a lot of work to do.

im also planning on writing a little guide to help anyone who wants to try out suckless core project in the future, but i have to say it is pretty easy to figure it out on your own.

i got carried away with this comment so heres everything i learnt when i was messing with device handling. This may prove useful to you if you ever want a graphical session in your vm where device handling actually matters.

/dev is automatically populated and mounted as a tempfs by default. Youd need to explicitly recompile the kernel without /dev tmpfs for this not to happen. Device handlers now are mainly for adjusting the permissions of a given device as they will automatically be created with owernership 'root:root'.

What i found is that smdev is built around a deprecated kernel feature that would expose a device called '/sys/kernel/hotplug'. The new uevent mechanism sends a message over a netlink socket whenever you hotplug a device. The information in that message can then be read by a daemon like busybox mdev with -d flag or udevd.

Busybox has a tool called uevent that you can run and it will spit these uevent messages out in plain text. I piped it to a shellscript and made a decent smdev-like setup using that. Perhaps one could make a service script that fires smdev -s whenever a uevent occurs?
