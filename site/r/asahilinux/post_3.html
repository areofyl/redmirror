I enjoy AI too but please stop wasting people's time with slop

by fake_agent_smith | 50 points

I'm not a member of the Asahi project, but this annoyed me so much I decided to try to draw some attention to the problem. This post is addressed to people who consider "helping" the project by prompting the gen AI for "solutions".

More and more people (or maybe even not people) recently started spamming the IRC with useless slop that provides no value and just wastes time. First and foremost, usage of AI is strictly forbidden by the Asahi project AI policy. It's entirely possible that some ideas coming from gen AI might have some merit, but at the same it's too much of a risk that the model might include some confidential data in the corpus, which could trigger lawsuit and basically kill Linux on Apple hardware. So it's not just stupid, but also possibly harmful to the project itself and to the wellbeing of project maintainers. So if you truly care about this project do not use gen AI for the contributions that might land in mainline. I personally disagree with some points in the policy but this is imo the important part:

Asahi Linux is a highly specific project, working in esoteric problem spaces on publicly undocumented hardware. Given the techniques used by Slop Generator manufacturers, it is not impossible for them to have confidential or leaked material owned by Apple or its vendor partners in their training corpi. It is therefore likely that Slop Generators will regurgitate this when queried in just the right way. We already forbid the use of illegally acquired or leaked documentation and tooling (e.g. Apple's internal repair diagnostic tools). This also applies to regurgitated slop.

FOSS projects like Asahi Linux cannot afford costly intellectual property lawsuits in US courts. The current political situation in that nation also makes it incredibly unlikely that any FOSS project would win such a suit regardless of the quality of its defence.

Hopefully this little explanation will reach the eyes of at least some people who need to see this.

adh1003 2

If I understand it’s limitations and review the code is really that bad?

A secondary answer on this part.

Ordinarily, you would write the code and someone else reviews it. The code can't contain anything you don't understand - it can't contain API calls you didn't know exist, etc.; you'd have Googled if you hit a wall, maybe checked StackOverflow, whatever.

Now you have an LLM generating usually a quite high volume of convincing-looking code which has a solid chance of using convincing-looking API calls you haven't investigated yourself. Do you read the docs on those? Understand if the error conditions or other boundary conditions are being dealt with properly, if the input ranges are right, if it's deprecated and old versus modern, whatever? Do you do that every single time?

And after all that, when you've finished reviewing your own code and babysitting an LLM - precisely because you know its limitations and realise just how careful you have to be - the best part is that someone else gets to review it all too.

Only normally, when we review someone else's code, we assume they've done some basic thinking around the problem space. Considered the other implications for the business and codebase based on their wider knowledge. It varies by seniority. Good reviewers account for all that.

Except you didn't; you gave it to an LLM. No reviewer can account for that; it's a completely unknown quantity. We don't know how much you actually thought about the problem or how much due diligence you did.

So now we either say "Yes, approved" or actually have to do your entire piece of work from start to finish, understanding the problem domain and possible solutions completely ourselves in order to effectively review your code.

Arguably, at that point, peer review is useless. You already did it. But of course, the lacking part here was that "thinking about the problem domain" part. There's just this crucial missing piece in that whole review story that's not the coding part but is at least, if not more important. EDIT: And of course, instead of increasingly-senior devs writing code with their increasingly-senior skills and domain knowledge, then independently peer reviewed by more-skilled devs, we instead have increasingly-senior devs writing code that's consistently at best junior grade, and then perhaps not even reviewed by anyone more senior at all. The coding quality doesn't lift with experience in the same way.

So that's why it can be bad. Like any powerful tool, it can be misused but AI seems to be able to be misused much more easily and accidentally than many other powerful tools.
